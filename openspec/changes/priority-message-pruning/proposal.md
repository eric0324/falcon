# Proposal: 基於優先度的訊息修剪

## 概述

現行的 auto compact 在觸發後會「機械地保留最後 6 則訊息 + 摘要」。這個策略簡單但會丟掉一些早期的關鍵訊息（例如第一則使用者需求描述），或保留了一堆中間的重複確認。此 change 在 compact 的基礎上加入「訊息優先度評分」：高優先度訊息（系統決策、工具結果、使用者重要指令）優先保留，低優先度（單純確認、冗長重複）優先被摘要或丟棄。

## 動機

- `src/lib/ai/compact.ts` 目前將舊訊息全摘要、只保留最後 6 則
- 不區分「重要訊息」和「填充對話」，在多工具步驟的對話中浪費 context
- 一些重要的工具呼叫結果（例如查詢到的關鍵數據）被摘要後精度下降
- 實際上有些早期訊息（使用者的初始需求）比中段對話更重要

## 目標

1. 為每則訊息計算 priority score（0-100）
2. Compact 時，依 priority + recency 雙軸決定保留/摘要
3. 高優先度訊息（使用者第一則需求、有 tool result 的、有附件的）優先保留原文
4. 低優先度訊息（純確認「好的」「謝謝」、重複內容）優先摘要
5. 保持現有的 summary 持久化機制

## 非目標

- 不用 AI 判斷優先度（太貴），用 heuristics
- 不讓使用者手動標記「pin this message」（未來可加）
- 不改變 compact 的觸發閾值（仍是 context 80%）
- 不改前端顯示邏輯（UI 仍顯示完整歷史）

## 優先度啟發式

**基礎分 50**，依以下加減：

| 條件 | 加分 |
|------|------|
| 對話第 1 則 user message（定錨需求）| +40 |
| 帶附件 | +25 |
| 包含 tool call result 的 assistant message | +20 |
| 長度 > 500 字元（實質內容） | +15 |
| 最近 3 則訊息 | +30 |
| 問號結尾（可能是重要提問） | +10 |

| 條件 | 減分 |
|------|------|
| 長度 < 20 字元 | -20 |
| 純確認詞「好」「ok」「謝謝」| -30 |
| assistant 的 streaming placeholder | -10 |

最終保留策略：
1. **必保留**：score >= 80 的全原文保留
2. **中段摘要**：40-79 的，按順序摘要
3. **優先丟棄**：< 40 的，直接從摘要輸入中剔除

## 風險

| 風險 | 緩解措施 |
|------|----------|
| Heuristic 誤判丟掉重要訊息 | 保守邊界（score >= 80 必保留）、保留最後 3 則不管分數 |
| 對話連貫性被破壞（中段被挖空） | 摘要的部分仍然覆蓋全部被摘要訊息，不是刪除 |
| 增加複雜度難以維護 | 純函數實作，單元測試覆蓋率要求 90%+ |
| 影響 auto-compact 既有行為 | 以 feature flag 方式導入，可快速 rollback |

## 驗收標準

1. 10 輪對話（全部短確認）觸發 compact → 大部分訊息被摘要，只保留第 1 則需求 + 最後 3 則
2. 10 輪對話（中段有大量 tool calls）→ tool results 保留原文，其他摘要
3. 帶附件的訊息即使在中段也不被丟棄
4. Compact 後的訊息順序正確、AI 能理解脈絡
5. 既有 compact 單元測試不破壞
6. 新增單元測試 ≥ 20 個 case
